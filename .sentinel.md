# Sentinel: Development Best Practices

> Evergreen principles learned from real bugs. These prevent classes of issues, not prescribe specific solutions.

---

## 🔄 State Management in SSR/Client Environments

### The Hydration Problem

Next.js (and similar frameworks) render on server first, then hydrate on client. State that exists server-side may not be immediately available client-side.

**Pattern: Use a `mounted` guard for client-only state**

```typescript
const [mounted, setMounted] = useState(false);
useEffect(() => setMounted(true), []);

// Only trust client-side state after mounting
if (!mounted) return <LoadingState />;
```

**Why it matters:** Without this, you get hydration mismatches where the server renders one thing and the client expects another.

---

### Multiple Loading States

When a feature depends on multiple async operations (e.g., session + role + data), gate on ALL of them:

```typescript
// ❌ BAD: Only checks one loading state
if (loading) return null;
if (isAdmin) showAdminUI();

// ✅ GOOD: Waits for everything to resolve
if (loading || roleLoading) return null;
if (isAdmin) showAdminUI();
```

**Why it matters:** Partial loading states cause UI to flash incorrect content before all dependencies resolve.

## ⚡ useEffect Dependencies

### The Stale Closure Problem

Effects with incomplete dependencies create stale closures. If an effect reads state, that state should be in deps.

```typescript
// ❌ BAD: Empty deps means this only runs once, even if isAdmin changes later
useEffect(() => {
  if (isAdmin) drawAdminIcon();
}, []);

// ✅ GOOD: Effect re-runs when relevant state changes
useEffect(() => {
  if (isAdmin) drawAdminIcon();
}, [isAdmin]);
```

**Canvas and animation components are especially vulnerable** - they draw once on mount and never update if deps are missing.

---

### API Call Storms

Multiple effects calling the same API create race conditions and unnecessary load.

**Pattern: Single-source-of-truth**

```typescript
// ❌ BAD: Three places calling the same API
useEffect(() => { fetchUserRole(); }, []);
useEffect(() => { if (user) fetchUserRole(); }, [user]);
onAuthStateChange(() => { fetchUserRole(); });

// ✅ GOOD: One effect owns the API call
useEffect(() => {
  if (user) fetchUserRole();
}, [user]);
```

**Principles:**
- Identify ONE effect responsible for each API call
- Other effects should read from state, not trigger new fetches
- Use refs or flags if you need to prevent duplicate calls during initialization

## 🔐 Async Operations

### The Null Check Narrowing Issue

TypeScript doesn't maintain null narrowing across await boundaries:

```typescript
// ❌ BAD: TypeScript can't guarantee client is still non-null after await
const client = getClient();
if (!client) return;
const data = await client.getData(); // TS error possible

// ✅ GOOD: Capture in const immediately after check
const client = getClient();
if (!client) return;
const safeClient = client; // Now safely narrowed
const data = await safeClient.getData();
```

**Why it matters:** In async functions, control flow analysis resets after `await`. Capture narrowed values in local constants.

## 🎨 Component Patterns

### Conditional Rendering with Multiple Conditions

When a component depends on multiple conditions, be explicit about the "not ready" states:

```typescript
// Pattern for auth-dependent UI
{mounted && !loading && !roleLoading && isAdmin && (
  <AdminButton />
)}
```

**Why it matters:** Each condition represents a potential state where the component shouldn't render. Being explicit prevents flash-of-wrong-content.

---

### Event Handler Timing

For dropdown menus and similar UI, be aware of event order: `mousedown → blur → click`

```typescript
// If dropdown closes on blur, the click might not fire
// Use onMouseDown for critical actions, or prevent blur
onMouseDown={(e) => {
  e.preventDefault(); // Prevents blur from firing
  handleAction();
}}
```

**Why it matters:** Click events fire after blur. If blur closes the dropdown, the click target disappears before the click completes.

## 🎯 CSS Considerations

### 3D Depth Over Visual Effects

When creating layered UI elements, use 3D transforms instead of visual effects like blur:

```typescript
// ❌ BAD: Blur obscures content and reduces visibility
<div style={{
  filter: 'blur(8px)',
  backdropFilter: 'blur(4px)',
}} />

// ✅ GOOD: 3D transforms create depth without hiding information
<div style={{
  transform: `translate3d(${x}px, ${y}px, ${z}px) scale(${scale})`,
  perspective: '1000px', // Enable 3D transforms
}} />
```

**Why it matters:** Blur filters obscure content and reduce usability, especially for interactive elements. 3D transforms (`translateZ`, `scale`) create natural depth perception without hiding information, enable GPU acceleration, and maintain content visibility.

### Visual Hints Over Full Rendering

For visual indicators that don't require full content, use CSS-only styling instead of full component rendering:

```typescript
// ❌ BAD: Rendering full components with images/videos for visual hints
{items.map(item => (
  <EntityCard entity={item} /> // Loads images, videos, etc.
))}

// ✅ GOOD: CSS-only visual hints
{Array.from({ length: count }).map((_, idx) => (
  <div style={{
    border: '1px solid rgba(...)',
    background: 'rgba(...)',
    opacity: 0.15,
    // No image/video loading
  }} />
))}
```

**Why it matters:** Visual hints provide the same user feedback (indicating additional content exists) without loading images/videos. This reduces bandwidth, GPU memory, and render complexity. When displaying many items (like in a list or grid), this optimization compounds significantly.

### Cross-Browser Support

Always include vendor prefixes for newer CSS features:

```css
backdrop-filter: blur(12px);
-webkit-backdrop-filter: blur(12px); /* Safari */
```

### Z-Index Layering

Document z-index values when they matter:

```css
/* z-index layers:
   0: base content
   1: overlays, tooltips
   100: sticky elements
   1000+: navigation, modals */
```

**Why it matters:** Undocumented z-index creates "z-index wars" where developers keep incrementing to fix stacking issues.

## ⚙️ Build-Time Configuration

### External Domain Whitelisting

Some Next.js features require build-time configuration that cannot be changed at runtime:

```typescript
// ❌ BAD: Trying to fix at runtime (won't work)
// API route or component code cannot fix this
export async function POST() {
  // This won't help - images.remotePatterns is build-time only
  return { success: true };
}

// ✅ GOOD: Configure at build time in next.config.ts
// next.config.ts
const nextConfig = {
  images: {
    remotePatterns: [
      {
        protocol: 'https',
        hostname: '*.supabase.co',
        pathname: '/storage/v1/object/public/**',
      },
    ],
  },
};
```

**Why it matters:** 
- Build-time config (like `images.remotePatterns`, `env` variables) must be set before deployment
- Runtime code changes cannot fix build-time configuration issues
- Always verify build configuration when debugging issues that persist after code changes
- Check Vercel/build logs for configuration-related errors

**Common build-time configurations:**
- `images.remotePatterns` - External image domains
- `env` variables - Environment-specific settings
- `experimental` features - Framework feature flags
- `redirects` / `rewrites` - URL routing rules

## 🚀 Vercel Deploy Guardrails (Hard-Learned)

These are the recurring “deploy gets rejected” failure modes we’ve hit in this repo.

### JSX Apostrophes / Quotes (ESLint: `react/no-unescaped-entities`)

Never put raw apostrophes/quotes in **JSX text nodes**. Vercel runs `next build` and will fail on ESLint errors.

```tsx
// ❌ BAD
<p>don't repeat it</p>

// ✅ GOOD (entities)
<p>don&apos;t repeat it</p>

// ✅ GOOD (JS expression)
<p>{"don't repeat it"}</p>
```

**Why it matters:** This is easy to miss locally and becomes a hard Vercel build failure.

---

### Prisma Query Args (TypeScript: “Please either choose `select` or `include`.”)

Prisma does **not** allow `select` and `include` together.

```ts
// ❌ BAD
await prisma.project.findFirst({
  where: { id: projectId },
  select: { id: true },
  include: { members: true },
})

// ✅ GOOD (nested select)
await prisma.project.findFirst({
  where: { id: projectId },
  select: { id: true, members: { select: { role: true } } },
})
```

---

### Prisma Schema Safety (The Big One)

**Do not run `prisma db pull --force` and commit the resulting `prisma/schema.prisma` changes.**

This repo’s Prisma schema is **hand-maintained** with `@map` / `@@map` to keep:
- **camelCase fields** (e.g. `createdAt`) while mapping to DB snake_case (e.g. `created_at`)
- **stable model names** (e.g. `Profile`, `Project`, `Output`)

**Failure pattern:** If `db pull` output is committed, TypeScript failures cascade everywhere:
- `prisma.profile` becomes `prisma.profiles`
- `createdAt` becomes `created_at`
- Vercel fails on “Property X does not exist…”

**Preferred ways to test DB connectivity (safe):**
- `prisma generate`
- `prisma migrate status`
- A small API “health check” query using the existing Prisma client

---

### Serverless DB Connections (Supabase + Vercel)

If you see errors like “Too many database connections open”, it usually means Vercel is using a **direct DB connection** instead of a pooler.

**Best practice:**
- Use Supabase **Connection Pooler** (port `6543`) for `DATABASE_URL` in **Vercel**.
- Add `?pgbouncer=true` for Prisma compatibility.

**Important:** Updating local `.env` does **not** update Vercel. You must update Vercel project env vars and redeploy.

---

### Tailwind Ambiguous Utility Warnings

Tailwind can warn that arbitrary classes are ambiguous (e.g. `ease-[cubic-bezier(...)]`). Prefer built-in utilities:
- `ease-in-out`, `ease-out`, etc.

If you truly need a custom curve, prefer inline style:

```tsx
<div style={{ transitionTimingFunction: 'cubic-bezier(0.4, 0, 0.2, 1)' }} />
```

## 🎬 Media Type Handling

### Graceful Fallback Chains

When displaying media, always handle the fallback chain explicitly. Different media types (images, videos) have different component requirements.

```typescript
// ❌ BAD: Assuming all media URLs work with Image component
const mediaUrl = video?.url || image?.url;
<Image src={mediaUrl} />  // Breaks if mediaUrl is a .mp4

// ✅ GOOD: Explicit fallback for different media states
const imageUrl = isVideo ? thumbnailUrl : rawMediaUrl;
const shouldRenderVideo = isVideo && !thumbnailUrl && rawMediaUrl;

{shouldRenderVideo ? (
  <video src={rawMediaUrl} autoPlay muted loop />
) : imageUrl ? (
  <Image src={imageUrl} />
) : (
  <Placeholder />
)}
```

**Why it matters:** 
- Next.js `Image` component cannot render video files (fails silently with broken image)
- Videos without thumbnails need explicit video element fallback
- Always consider: What if the expected media format isn't available?

### Media Persistence

When uploading videos, ensure thumbnails are extracted and persisted:

```typescript
// During upload: extract thumbnail from video
const thumbnailBlob = await extractVideoThumbnail(videoFile);
const thumbnailUrl = await uploadThumbnail(thumbnailBlob);

// Persist BOTH video URL and thumbnail URL to database
await saveEntity({ videoUrl, thumbnailUrl });
```

**Why it matters:** Runtime thumbnail extraction causes CORS issues during export. Pre-extracted thumbnails enable reliable PNG exports and faster card rendering.

### Next.js Image Optimization Domain Whitelisting

When using Next.js `Image` component with external image sources (Supabase Storage, S3, CDNs), you must whitelist those domains in `next.config.ts`:

```typescript
// ❌ BAD: External images fail with 400 errors, no visible error in UI
// next.config.ts has no images configuration
<Image src="https://project.supabase.co/storage/v1/object/public/bucket/image.jpg" />

// ✅ GOOD: Whitelist external domains in next.config.ts
// next.config.ts
const nextConfig = {
  images: {
    remotePatterns: [
      {
        protocol: 'https',
        hostname: '*.supabase.co',
        pathname: '/storage/v1/object/public/**',
      },
      // Add other external domains as needed
    ],
  },
};
```

**Why it matters:** 
- Next.js Image Optimization requires explicit domain whitelisting for security
- Without whitelisting, images fail silently with 400 errors (only visible in console)
- This is a **build-time configuration** - must be set before deployment
- Runtime fixes (cache revalidation, URL updates) cannot resolve this issue

**Pattern for common services:**
- **Supabase Storage**: `*.supabase.co` with path `/storage/v1/object/public/**`
- **AWS S3**: `*.s3.amazonaws.com` or `*.s3.*.amazonaws.com`
- **Cloudinary**: `res.cloudinary.com`
- **Custom CDN**: Match your CDN domain pattern

## 🔌 External API Integration

### Silent Parameter Failures

External APIs (especially ML/AI APIs like Replicate) often **silently ignore** unknown parameters instead of returning errors. This creates hard-to-debug issues where features appear to work but don't.

```typescript
// ❌ BAD: Wrong parameter name - API silently ignores it
input.image_urls = referenceImages;  // This param doesn't exist!
// API accepts request, generates text-to-image instead of image-to-image
// No error, no warning - just wrong output

// ✅ GOOD: Correct parameter name with explicit logging
input.image_input = referenceImages;
console.log(`[API] ✅ Using ${referenceImages.length} reference image(s)`);
console.log(`[API] First image type: ${referenceImages[0]?.startsWith('data:') ? 'data URL' : 'URL'}`);
```

**Why it matters:**
- REST APIs don't have type-safety across boundaries
- Unknown/misspelled parameters are silently dropped
- The request "succeeds" but produces wrong results
- Users see their input in the UI but it has no effect

**Prevention Pattern: Explicit "is using" vs "not using" logging**

```typescript
// Always log BOTH cases explicitly
if (referenceImages.length > 0) {
  input.image_input = referenceImages;
  console.log(`[Model] ✅ Using ${referenceImages.length} reference image(s)`);
} else {
  console.log(`[Model] ⚠️ No reference images - text-to-image only`);
}
```

**Why this helps:**
- During testing, you can verify the "✅ Using" log appears
- If output is wrong despite the log, the bug is in API param name
- If log says "⚠️ No reference images", bug is in how images are passed to the function

### Verify Against Working Implementations

When implementing API integrations, always compare with existing working code:

```typescript
// ❌ BAD: Guessing parameter names from memory
// "I think it was image_urls..."

// ✅ GOOD: Copy exact parameter names from working implementation
// Look at replicate.ts Seedream 4.5 - it uses image_input
// Look at Replicate docs - they show exact param names
```

**Why it matters:** The same provider (e.g., Replicate) may use consistent parameter names across models. If Seedream uses `image_input`, Nano Banana Pro likely does too.

### Fetch Response Bodies Are Single-Use

In Fetch, a `Response` body is a **one-time stream**. Once you read it (`.json()`, `.text()`, etc.), it cannot be read again.

```typescript
// ❌ BAD: "Body has already been read"
const res = await fetch(url)
const json = await res.json()
const text = await res.text()

// ✅ GOOD: Read once, or clone before a second read
const res2 = await fetch(url)
const res2Copy = res2.clone()
const json2 = await res2.json()
const text2 = await res2Copy.text()
```

**Why it matters:** This often only shows up in “retry” paths (when you parse an error payload, then want to reuse the same response later), and it crashes otherwise-successful requests.

### Polling Endpoints: Payload Shape Drift (Object vs Array)

External status endpoints evolve. The same field can arrive as an **object** or an **array** depending on version, model, or provider-side rollout.

```typescript
const dataField = statusData.data

// Normalize the “task record” regardless of object vs array shape
const taskRecord = Array.isArray(dataField)
  ? dataField.find((item) => item && typeof item === 'object' && (item as any).task_id === taskId) ?? null
  : dataField && typeof dataField === 'object'
    ? dataField
    : null

const taskStatus = String(
  (taskRecord as any)?.task_status ??
    (taskRecord as any)?.status ??
    ''
).toLowerCase()
```

**Why it matters:** If your parser assumes one shape, you can end up “successfully polling forever” while never observing a terminal state — the UI gets stuck in “processing”.

### Server-Side Call Storms: Duplicate Background Processing

If multiple actors can trigger processing (server + client, retries, webhooks, etc.), you must make processing **idempotent** with an **atomic lock**.

```typescript
// ❌ BAD: Non-atomic read → check → write (racy)
const job = await prisma.job.findUnique({ where: { id } })
if (job?.startedAt) return { status: 'skipped' }
await prisma.job.update({ where: { id }, data: { startedAt: new Date() } })

// ✅ GOOD: Atomic compare-and-set
const locked = await prisma.job.updateMany({
  where: { id, startedAt: null },
  data: { startedAt: new Date() },
})
if (locked.count === 0) return { status: 'skipped' }
```

**Why it matters:** Duplicate processing creates flakiness, inflates cost, and makes external API issues appear “random”. Treat your DB as the single source of truth for “who owns this job right now”.

## 📊 Large Data & String Processing

### Regex on Large Strings Causes Stack Overflow

JavaScript's regex engine can blow the call stack when processing very large strings (e.g., 15MB+ base64 data URLs). This manifests as "Maximum call stack size exceeded" errors that are hard to trace.

```typescript
// ❌ BAD: Regex with greedy patterns on large strings
const matches = base64DataUrl.match(/^data:([^;]+);base64,(.+)$/)
// This WILL crash on large files (e.g., high-res images, videos)

// ✅ GOOD: Use indexOf/slice for parsing large strings
const commaIndex = base64DataUrl.indexOf(',')
if (commaIndex === -1) throw new Error('Invalid format')
const meta = base64DataUrl.slice('data:'.length, commaIndex)
const base64Data = base64DataUrl.slice(commaIndex + 1)
```

**Why it matters:**
- Regex engines use recursion internally for greedy patterns like `.+`
- A 15MB string can trigger millions of recursive calls
- The error "Maximum call stack size exceeded" appears far from the actual regex call
- `indexOf()` and `slice()` are O(n) without recursion - safe for any size

**Pattern: Prefer string methods over regex for data parsing**
- Use `indexOf()`, `slice()`, `split()` for structural parsing
- Reserve regex for small strings or pattern matching, not data extraction

## 🔄 API Response Type Coercion

### External APIs Return Unexpected Types

Different APIs return the same conceptual data in different types. A duration might be a number from one API and a string from another. This causes database write failures when Prisma expects a specific type.

```typescript
// ❌ BAD: Trusting API response types match your schema
await prisma.output.create({
  data: {
    duration: apiResponse.duration, // Might be "5.041" (string) not 5.041 (number)
  }
})
// Prisma error: "Expected Float, provided String"

// ✅ GOOD: Coerce types at the boundary
const coerceFloatOrNull = (value: any): number | null => {
  if (value === null || value === undefined) return null
  if (typeof value === 'number') return Number.isFinite(value) ? value : null
  if (typeof value === 'string') {
    const parsed = parseFloat(value)
    return Number.isFinite(parsed) ? parsed : null
  }
  return null
}

await prisma.output.create({
  data: {
    duration: coerceFloatOrNull(apiResponse.duration),
  }
})
```

**Why it matters:**
- External APIs have no contract with your database schema
- The same API might change response types between versions
- Type mismatches cause cryptic Prisma/DB errors
- Coercion functions document the expected transformation

## 🚫 API Feature Incompatibilities

### Mutually Exclusive Parameters

Some APIs have parameters that cannot be used together. This isn't always documented clearly and manifests as validation errors only at runtime.

```typescript
// ❌ BAD: Assuming all features can be combined
const input = {
  image: startFrame,
  image_tail: endFrame,  // Frame interpolation
  sound: 'on',           // Audio generation
}
// API error: "model/mode/duration is not supported with image_tail when sound is on"

// ✅ GOOD: Enforce known incompatibilities in code
if (input.image_tail) {
  // Kling API limitation: frame interpolation and audio are mutually exclusive
  if (input.sound === 'on') {
    input.sound = 'off'
    console.log('[API] Disabled sound for frame interpolation (API limitation)')
  }
}
```

**Why it matters:**
- API docs may not list all incompatibilities
- These errors only appear at runtime with specific parameter combinations
- Silently fixing incompatibilities (with logging) is better than failing
- Document discovered incompatibilities in code comments for future reference

**Pattern: Document incompatibilities as code comments with source**
```typescript
// Kling API limitation: kling-v2-6/pro/5s does not support image_tail with sound=on
// Discovered: 2024-01-12 via runtime error
// Source: https://app.klingai.com/global/dev/document-api
```

## 🏷️ API Version/Model Name Formats

### Version Naming Conventions Vary

APIs use different conventions for version names. Dots, hyphens, and underscores are NOT interchangeable.

```typescript
// ❌ BAD: Guessing version format
model_name: 'kling-v2.6'  // WRONG - uses dot
// API error: "model_name value 'kling-v2.6' is invalid"

// ✅ GOOD: Use exact format from API docs
model_name: 'kling-v2-6'  // CORRECT - uses hyphen

// ✅ BETTER: Add a comment with doc reference
// NOTE: Official docs use hyphenated version names (e.g. "kling-v2-6"), not dotted
// Source: https://app.klingai.com/global/dev/document-api/apiReference/model/imageToVideo
model_name: 'kling-v2-6',
```

**Why it matters:**
- Version strings are opaque identifiers to the API - must match exactly
- Common variants: `v2.6`, `v2-6`, `v2_6`, `2.6`, `26`
- Always verify against official API documentation
- Include doc link in comments for future reference

## Performance Postmortem: The Sigil Loading Crisis (March 2026)

> **What finally moved the needle:** Two fixes accounted for 95% of the improvement. (1) A one-line Vercel region config (`vercel.json` with `"regions": ["cdg1"]`) that co-located compute with the database, dropping every DB round trip from ~600ms to ~5ms. (2) Excluding oversized inline `data:image/...` base64 URIs (up to 29MB per row) from thumbnail queries and SSR prefetch payloads, which had been silently blocking server render for 5-15 seconds per page.

### Timeline of Attempts

Over ~10 days, the codebase went through roughly eight performance-oriented commits touching the same core files (`lib/prefetch/dashboard.ts`, `lib/prefetch/journeys.ts`, `app/dashboard/page.tsx`, `app/journeys/page.tsx`, `components/dashboard/DashboardView.tsx`). The progression:

1. **Server-side prefetch + parallel queries + dynamic imports** -- Correct direction. Reduced sequential DB calls from 3 to 2. Saved ~600ms per page on paper, but production still hit 20-30s loads.
2. **Eliminate prefetch storm + slim payloads + LATERAL SQL thumbnails** -- Reduced redundant queries and payload size. Marginal improvement locally, negligible on production.
3. **`force-dynamic` toggle** -- Added then removed. Prevented ISR but also broke caching; net neutral.
4. **SWR conversion for journeys + shared query logic + `/api/me` elimination** -- Removed duplicate client-side refetches and consolidated data paths. Good architectural cleanup, but production latency unchanged.
5. **Vercel region co-location (`cdg1`)** -- The breakthrough. Dropped production TTFB from 20-30s to 1-2s immediately.
6. **Conditional thumbnail loading (skip on SSR, revalidate on client)** -- Fixed local dev latency where thumbnail SQL was blocking page render for 5-15s.
7. **Data URI exclusion from thumbnail queries** -- The final fix. Eliminated 11-29MB base64 strings from DB result sets and server-rendered payloads.
8. **Lightweight workspace prefetch (skip output URLs, hydrate via SWR)** -- Cut route-open time from ~10s to ~1s by not serializing massive generation outputs into the RSC payload.

### Why Most Changes Underperformed

The fundamental issue was **optimizing the wrong layer**. The actual SQL execution time for even the most complex thumbnail query was 0.127ms. The database was idle. All latency came from:

- **Network round trips** (~600ms each, US compute to EU database)
- **Serial query chains** (3+ sequential awaits = 1800ms+ floor)
- **Payload size** (inline `data:` URIs up to 29MB inflating DB reads, JSON serialization, and RSC streaming)

Optimizing query structure, parallelization, and caching produced real architectural improvements but could not overcome the 600ms-per-hop physics. Each removed query saved one RTT; co-location made every remaining RTT negligible.

### The Decisive Evidence

Runtime instrumentation (`fs.appendFileSync` to an NDJSON log file) provided the proof:

- `prefetchDashboard` with thumbnails: `durMs.thumbs: 11631` (11.6 seconds in thumbnail stage alone)
- `prefetchWorkspaceData`: `dataQueries: 4851ms`, `maxOutputUrlChars: 29852690` (29MB in a single output URL)
- `prefetchJourneyDetail`: `dataUriThumbCount: 1`, `maxThumbUrlChars: 11666695` (11.6MB thumbnail URL)
- Auth stages: `0ms` (cached via `React.cache`), confirming auth was never the bottleneck

Without this instrumentation, the oversized `data:` URI payloads would not have been identified -- they produced no errors, no warnings, and no visible failures. They simply made every query that touched those rows take 5-15 seconds to serialize and transfer.

### What the Earlier Refactors Actually Achieved

The pre-fix refactors were not wasted. They produced lasting architectural improvements:

- **Parallel `Promise.all` for profile + data queries** -- Structural improvement that compounds with co-location
- **SWR with `fallbackData` for journeys** -- Eliminated duplicate client/server data fetches
- **Server-side auth pass to `AuthProvider`** -- Removed a client-side `/api/me` round trip on every navigation
- **Shared query logic between prefetch and API routes** -- Single source of truth for data shapes
- **`RequireAuth` replaced with server redirect** -- Faster auth check, no client-side gate

These changes hardened the architecture and will prevent regression as the app scales. They just couldn't overcome the 600ms/hop network physics.

### Portable Lessons

1. **Measure before optimizing.** Instrument the actual runtime, not the code structure. Without log evidence, we spent days optimizing query parallelization when the real bottleneck was network geography and payload size.
2. **Check infrastructure first.** Compute/data co-location is a prerequisite, not an optimization. If your app compute and database are in different continents, no amount of code optimization will fix latency.
3. **Watch for invisible payload bloat.** Inline `data:` URIs in database columns are invisible to code review. They look like normal strings until you measure their length at runtime. Always filter or cap URL lengths in list/thumbnail queries.
4. **One-layer-at-a-time changes with post-measurement.** Multi-layer changes (simultaneously tweaking SSR, SWR, cache headers, and query structure) make it impossible to attribute improvement. Change one variable, measure, keep or revert.
5. **Prefetch should be cheap.** Server-side prefetch exists to give the client a head start, not to do the client's full job. Keep prefetch payloads small and let client-side SWR hydrate the rest.

## Performance Architecture Patterns

### Latency Amplification

Small SQL + large network RTT + serial calls = multiplicative latency. A query that executes in 0.1ms can take 600ms wall-clock if the round trip dominates. Three such queries in series = 1800ms minimum, regardless of SQL optimization.

```
Apparent latency = (num_serial_queries * RTT) + (total_sql_time)

With co-location:  3 * 5ms  + 0.3ms =   15ms
Without:           3 * 600ms + 0.3ms = 1800ms
```

**Prevention:** Always know where your compute runs relative to your database. Verify with a simple ping measurement before optimizing queries.

### Dual-Fetch / Hydration Mismatch

When a server component prefetches data and passes it as `fallbackData` to a client-side SWR hook, misconfigured `revalidateOnMount` causes the client to immediately refetch the same data. This doubles effective load time.

```typescript
// BAD: SWR refetches on mount even though server already provided data
const { data } = useSWR("/api/data", fetcher, {
  fallbackData: serverData,
  // revalidateOnMount defaults to true when no fallbackData config
});

// GOOD: Skip mount revalidation when server data is complete
const { data } = useSWR("/api/data", fetcher, {
  fallbackData: serverData,
  revalidateOnMount: !serverData,
  revalidateOnFocus: false,
  dedupingInterval: 60_000,
});
```

**Variant -- partial prefetch:** When SSR intentionally skips expensive data (e.g., thumbnails), set `revalidateOnMount` to `true` so the client fills in the gap:

```typescript
revalidateOnMount: !initialData || !initialDataIncludesThumbnails,
```

### Payload Toxicity

Oversized values in database columns (inline base64 images, large JSON blobs, full file contents) silently poison every query that touches those rows. The query planner reports sub-millisecond execution, but transferring and serializing the results takes seconds.

```sql
-- BAD: Returns rows with 29MB file_url values
SELECT o.file_url FROM outputs o WHERE ...

-- GOOD: Exclude inline data URIs from list/thumbnail queries
SELECT o.file_url FROM outputs o
WHERE ... AND o.file_url NOT LIKE 'data:%'
```

**Prevention:**
- Never store large binary data as inline column values in rows that appear in list queries
- If inline storage is unavoidable, always filter in SQL (`NOT LIKE 'data:%'`) for list/thumbnail/prefetch paths
- Set payload size budgets: prefetch responses should be under 100KB; individual URL fields should be under 2KB

### Instrument-First Debugging

When performance is bad and the cause is unclear, do not guess. Instrument with runtime logging, reproduce, read the evidence, then fix.

```
1. Generate 3-5 hypotheses about WHERE time is spent
2. Add timing instrumentation to each suspect code path
3. Reproduce the issue and collect logs
4. Evaluate each hypothesis: CONFIRMED / REJECTED / INCONCLUSIVE
5. Fix only what is confirmed by evidence
6. Verify with a second instrumented run
7. Remove instrumentation after confirmation
```

**Key principle:** A hypothesis rejected by runtime evidence should have its code changes reverted. Do not accumulate speculative fixes from discarded theories.

## System Vulnerabilities to Watch

### Cross-Region Drift

App compute and primary data stores must be in the same region. This is not an optimization -- it is a prerequisite. Vercel defaults to `iad1` (US East). If your database is in EU, every query pays ~600ms.

**Detection:** Run `curl -w "TTFB: %{time_starttransfer}s"` against your production endpoint. If TTFB exceeds 3s for a page with simple queries, suspect region mismatch.

**Fix:** `vercel.json` with `"regions": ["cdg1"]` (or whichever region matches your DB). Verify with a deployment config audit at least once per quarter.

### Hidden Fan-Out

A single page navigation can trigger auth checks in middleware, layout, and page components -- each in a separate execution context that cannot be deduplicated by `React.cache`. Combined with client-side SWR revalidation and context provider initialization, a single click can produce 5-8 network round trips.

**Detection:** Count total DB calls per navigation in server logs. If > 3 for a read-only page, investigate.

**Mitigation:**
- Auth: resolve once in the outermost server component, pass down as props
- Data: one prefetch function per page surface, not per component
- Client: `revalidateOnMount: false` when server data is sufficient

### Unbounded Aggregation

Admin and analytics endpoints that load all rows and aggregate in JavaScript work at small scale but degrade linearly. A `findMany` without `take` or date filtering is a scaling time bomb.

**Rule:** Every list query must have either a `take` limit, a date range, or both. Aggregation should happen in SQL, not application code.

### Generation Pipeline Concurrency

If multiple actors (server, client, webhook, retry) can trigger the same background job, you need atomic locking. Non-idempotent processing causes duplicate API calls, inflated costs, and hard-to-trace "random" failures.

**Rule:** Use compare-and-set (`updateMany` with `where: { startedAt: null }`) for job claiming. If `count === 0`, someone else owns it.

## Guardrails (Scale-Ready)

### Route-Level Performance Budgets

| Surface | TTFB | p95 Total | DB Queries | Prefetch Payload |
|---------|------|-----------|------------|------------------|
| Dashboard | < 500ms | < 2s | <= 3 | < 100KB |
| Journeys list | < 500ms | < 2s | <= 3 | < 100KB |
| Journey detail | < 300ms | < 1s | <= 2 | < 50KB |
| Route workspace | < 500ms | < 2s | <= 3 | < 50KB (no outputs) |

### Single Owner Per Fetch

Each data fetch path has exactly one owner:
- **SSR prefetch** owns the initial page data (lightweight, no heavy media)
- **Client SWR** owns revalidation and heavy data hydration (thumbnails, outputs)
- **API route** owns the canonical data shape and caching headers
- No component should independently `fetch` data that another component already fetches

### No Inline Large Media in SSR/API Contracts

- Thumbnail queries must include `AND file_url NOT LIKE 'data:%'`
- SSR prefetch must not include generation outputs with potentially large `fileUrl` values
- API routes returning lists must cap per-item payload size or exclude blob fields

### Measurement Before/After for Performance PRs

Every PR that claims a performance improvement must include:
- Before measurement (with methodology: curl TTFB, dev server logs, or runtime instrumentation)
- After measurement using the same methodology
- Explicit keep/revert decision based on the delta

## Multi-User Stability Roadmap

### Immediate (0-2 weeks)

**Observability and alerting:**
- Add `Server-Timing` headers to all API routes (already partially done; standardize)
- Set up p95 latency alerting on Vercel Analytics for dashboard, journeys, route-open
- Add correlation IDs to middleware that propagate through API calls and DB queries

**Bounded queries:**
- Audit all `findMany` calls without `take` -- add limits or pagination
- Move admin dashboard-stats and cost-report aggregation to SQL `GROUP BY`
- Add date-range parameters to analytics endpoints

**Auth efficiency:**
- Ensure middleware uses `getSession()` (cookie-only, no DB call) instead of `getUser()` for route protection
- Reserve `getUser()` (DB-verified) for server components that need profile data

### Medium-Term (2-6 weeks)

**Generation pipeline hardening:**
- Implement idempotency keys for generation requests (prevent duplicate API calls on retry)
- Add atomic job claiming with `updateMany` compare-and-set
- Introduce per-user/per-project rate limiting (backpressure) for generation submissions
- Dead-letter queue for failed generations with admin visibility

**Data architecture:**
- Migrate inline `data:` URI outputs to object storage (Supabase Storage or S3) with URL references
- Generate thumbnail derivatives asynchronously (resize, WebP conversion) and cache on CDN
- Split generation list queries: metadata-only for lists, full output data on detail view

**Caching strategy:**
- Edge caching for public/demo pages (Vercel Edge Middleware + `Cache-Control`)
- Private short-TTL caching for authenticated pages (`Cache-Control: private, max-age=10, stale-while-revalidate=30`)
- Consider `unstable_cache` or in-memory TTL cache for server-side prefetch results

### Long-Term (6+ weeks)

**Operational maturity:**
- Synthetic monitoring probes for key user flows (dashboard -> journey -> route -> generate)
- Load testing for multi-tenant peak behavior (10+ concurrent users generating)
- Performance regression gate in CI: fail PR if lighthouse score drops or bundle size increases beyond threshold

**Infrastructure scaling:**
- Evaluate Prisma Accelerate for edge connection pooling and query caching
- Consider read replicas if analytics queries compete with transactional writes
- Evaluate Supabase Realtime for generation status instead of polling

## React / Next.js Best Practice Alignment

> Aligned with [Vercel Engineering React Best Practices v1.0.0](https://github.com/vercel-labs/agent-skills/tree/main/skills/react-best-practices) (January 2026).

### Eliminating Waterfalls (CRITICAL)

Waterfalls are the single largest performance killer. Each sequential `await` adds full network latency.

**Rules to enforce:**
- Use `Promise.all()` for independent data fetches in server components and API routes
- Defer `await` until the branch where the result is actually needed
- Use component composition to parallelize data fetching (each server component fetches its own data concurrently)
- Place `<Suspense>` boundaries around slow data components so fast content streams immediately

```typescript
// BAD: sequential (each await blocks the next)
const profile = await getProfile(userId);
const projects = await getProjects(userId);
const stats = await getStats(userId);

// GOOD: parallel (all three start simultaneously)
const [profile, projects, stats] = await Promise.all([
  getProfile(userId),
  getProjects(userId),
  getStats(userId),
]);
```

### Minimize Serialization at RSC Boundaries (HIGH)

Server components pass props to client components via JSON serialization. Large objects (generation outputs with base64 URLs, deeply nested data) inflate the RSC payload and slow streaming.

**Rules to enforce:**
- Only pass the minimum data a client component needs
- Never pass full database records; select and reshape first
- Flag fields that could contain large values (file URLs, JSON blobs) and exclude from SSR props

### SWR Contracts (MEDIUM-HIGH)

Every client component that uses SWR must document its revalidation contract:

```typescript
// Contract: DashboardView
// - fallbackData: lightweight server prefetch (no thumbnails)
// - revalidateOnMount: true (hydrate full data including thumbnails)
// - revalidateOnFocus: false (user-triggered refresh only)
// - dedupingInterval: 60s (prevent duplicate fetches within window)
```

### Re-render Control (MEDIUM)

- Calculate derived state during rendering instead of in `useEffect` + `setState`
- Use `useTransition` for non-urgent updates (admin stats, analytics)
- Extract frequently-updating components into memoized sub-components
- Use functional `setState` updates to avoid stale closure bugs
- Narrow `useEffect` dependencies to the specific values that should trigger re-execution

### Suspense and Streaming (MEDIUM)

Place `<Suspense>` boundaries strategically so the shell renders immediately while data-dependent sections stream in:

```tsx
<NavigationFrame>
  {/* Shell renders instantly */}
  <Suspense fallback={<DashboardSkeleton />}>
    {/* Data-dependent content streams when ready */}
    <DashboardContent />
  </Suspense>
</NavigationFrame>
```

### Bundle Hygiene (MEDIUM)

- Use `next/dynamic` for heavy components not needed on initial render (modals, editors, canvas)
- Avoid barrel file imports (`import { x } from '@/components'`) -- import directly from the file
- Preload components on user intent (hover/focus) with `dynamic(() => import(...), { ssr: false })`
